# AI Agent Guidelines

## Repository Overview

This is a **documentation-only repository** (`ruzickap/cks-notes`)
containing Certified Kubernetes Security Specialist study notes. The
primary content is `README.md` (Markdown with embedded shell/YAML code
blocks). There is no application source code, build system, or test
framework.

## Validation Commands

CI runs via MegaLinter (documentation flavor). There is no build or
test suite -- linting **is** the test suite.

```bash
# Markdown linting (primary validation tool)
rumdl README.md
rumdl AGENTS.md

# Shell validation -- extract bash blocks from Markdown, then lint
# (CI uses `mq` to extract; manually check individual blocks)
shellcheck --exclude=SC2317 script.sh
shfmt --case-indent --indent 2 --space-redirects --diff script.sh

# Link checking
lychee --config lychee.toml README.md

# JSON validation (if JSON files are added)
jsonlint --comments file.json

# GitHub Actions workflow validation
actionlint
```

To run the full CI suite locally, use the MegaLinter Docker image:

```bash
docker run --rm \
  -v "$(pwd):/tmp/lint" \
  oxsecurity/megalinter-documentation:v9
```

## Markdown Style

- **Line length**: Wrap prose at 72 characters; code blocks are exempt
  (configured in `.rumdl.toml`: `MD013.code-blocks = false`)
- **Heading hierarchy**: Never skip levels (H1 -> H2 -> H3, etc.)
- **Code fences**: Always include a language identifier:
  - `bash` for executable commands
  - `console` for command + output examples (use `$` prompt)
  - `yaml`, `json`, etc. for data formats
- **Indentation**: Two spaces, never tabs
- **Excluded files**: `CHANGELOG.md` is excluded from all linting
  (auto-generated by Release Please)

## Shell Code Blocks in Markdown

Bash/shell/sh code blocks are **extracted from Markdown and validated
in CI** using `shellcheck` and `shfmt`. Treat them as real scripts:

- **Variables**: Use uppercase with braces: `${MY_VARIABLE}`
- **Indentation**: 2 spaces (enforced by `shfmt --indent 2`)
- **Redirects**: Space before operator (`cmd > file`, not `cmd >file`)
- **Case statements**: Indent case bodies (`shfmt --case-indent`)
- **Suppressed warning**: SC2317 (unreachable command) is excluded
- **Heredocs**: Use `<< EOF ... EOF` for multi-line YAML/config

## Link Checking (lychee)

Configuration lives in `lychee.toml`:

- **Accepted status codes**: 200 and 429 (rate-limited)
- **Caching**: Enabled; 403/429 responses are re-checked
- **Excluded URLs**: Template variables (`%7B.*%7D`), shell
  variables (`\$`)
- **Excluded paths**: `CHANGELOG.md`, `package-lock.json`
- **Private IPs**: All private addresses are excluded

## Security Scanning

Multiple overlapping scanners run in CI:

- **Checkov**: IaC scanner (skips `CKV_GHA_7`)
- **DevSkim**: Pattern scanner (ignores DS162092, DS137138;
  excludes `CHANGELOG.md`)
- **KICS**: Fails only on HIGH severity
- **Trivy**: HIGH/CRITICAL only, ignores unfixed vulnerabilities
- **Gitleaks**: Secret detection (allowlists `README.md`)
- **CodeQL**: Analyzes GitHub Actions workflows
- **OSSF Scorecard**: Security best practices audit

## GitHub Actions Conventions

When modifying `.github/workflows/*.yml`:

- **Validate**: Run `actionlint` after every change
- **Permissions**: Use `permissions: read-all` at workflow level
- **Pin actions**: Always use full SHA with semver comment:
  `uses: actions/checkout@<full-sha> # v6.0.2`
- **Timeouts**: Set `timeout-minutes` on every job (5 or 10)
- **YAML format**: Start files with `---` and a descriptive comment
- **Ordering**: Use `# keep-sorted` markers for alphabetically sorted blocks
- **Triggers**: Include `workflow_dispatch` on every workflow
- **Runners**: Prefer `ubuntu-24.04-arm` where tools support it

## Commit Messages

Conventional commit format, validated by `commit-check` action.

- **Format**: `<type>: <description>`
- **Types**: `feat`, `fix`, `docs`, `chore`, `refactor`, `test`,
  `style`, `perf`, `ci`, `build`, `revert`
- **Subject**: Imperative mood, lowercase, no trailing period,
  max 72 characters
- **Body**: Optional; wrap at 72 chars, explain what and why
- **Issue refs**: Use `Fixes`, `Closes`, or `Resolves`

```text
docs: add network policy examples for CKS exam

- Add ingress/egress policy templates
- Include namespace isolation patterns

Resolves: #42
```

## Branching

Follow the [Conventional Branch](https://conventional-branch.github.io/)
specification: `<type>/<description>`

- `feature/` or `feat/`: New features
- `bugfix/` or `fix/`: Bug fixes
- `hotfix/`: Urgent fixes
- `release/`: Release prep (e.g., `release/v1.2.0`)
- `chore/`: Non-code tasks (dependency updates, docs)

Use lowercase, hyphens, no consecutive/leading/trailing hyphens.
Include ticket numbers when applicable: `feat/issue-42-add-rbac-notes`

## Pull Requests

- **Always create as draft** initially
- **Title**: Must follow conventional commit format (validated by
  `semantic-pull-request` action)
- **Description**: Explain changes and motivation clearly
- **Link issues**: Use `Fixes`, `Closes`, or `Resolves` keywords
